Vibe Coding で個人ブログを作ってみました。

技術的にはそこまで難しいことはしていません。  
Markdownで記事を書いて、Gitで管理して、HTMLに変換して、
GitHub Pagesで公開するだけです。

正直に言うと、  
この構成自体は以前から「やろうと思えばできる」ものでした。  
ただ、簡単だけど地味に面倒で、作る気にはなれていませんでした。

最近、AIでコードを書くことが当たり前になり、  
こうした準備や整備のコストが一気に下がりました。

「今なら、さっと作れるな」と思ったのが、  
このブログを作ったきっかけです。

---

## 技術的な構成について

このブログは、以下のような構成で運用しています。

- Markdownで記事を書く
- Gitで履歴を管理する
- GitHub Actionsでビルド
- GitHub Pagesで公開

記事を書いて push すれば、自動的に公開されます。  
ブログとしてはかなりシンプルで、  
日常の開発作業の延長で扱えるのが気に入っています。

VSCodeなどのIDE上ですべて完結するため、  
特別なツールを意識する必要もありません。

---

## Knowledge as Code という考え方

このブログは、**Knowledge as Code** という考え方を試す場でもあります。

システムエンジニアにとっての知識の資産は、  
文章だけでなく、コードや設計、判断の履歴も含めたものだと思っています。

Gitで管理することで、

- どこをどう直したか
- いつ考えが変わったか
- 何を試して、何をやめたか

といった情報が、差分として自然に残ります。

これは、  
知識を「書き捨てるもの」ではなく、  
**コードと同じように育てていくもの**として扱える、  
という点でしっくり来ています。

---

## AI時代には、より相性が良いと感じている

AIが前提になると、  
コードを書くこと自体のコストは大きく下がります。

その一方で、

- なぜその構成にしたのか
- どこを割り切ったのか
- 何を前提に判断したのか

といった、**コードの外側にある情報**の重要性は、  
むしろ上がっていると感じています。

Markdown + Git + IDE という構成は、  
AIとの相性も良く、文章を書きながら考えを整理したり、  
対話しながら内容を詰めたりするのがとても楽です。

AIで実装が速くなるほど、  
こうした知識の置き場が重要になる気がしています。

---

## マネジメント寄りになってきて感じていること

最近は、実装よりもマネジメント寄りの仕事が増え、  
部下や後輩と関わる時間も多くなってきました。

そうなると、  
自分がこれまで当たり前のようにやってきたことや、  
頭の中で暗黙的に判断していたことを、  
**きちんと形にして伝える必要がある**と感じる場面が増えてきます。

その場で口頭で説明するだけではなく、

- なぜそう考えているのか
- どういう前提で判断しているのか
- 失敗した経験も含めてどう整理しているのか

といった内容を、  
あとから参照できる形で残しておきたい、  
という気持ちが強くなりました。

このブログは、  
そうした知識や考えを整理し、  
誰かに渡せる形にするための置き場でもあります。

---

## ZennやQiitaとは「誰に書くか」が違う

ZennやQiitaは、今でもとても良い場所だと思っています。  
不特定多数に向けて、再利用しやすい知識を届けるには最適です。

一方で、このブログは、  
広く拡散することを目的にはしていません。

- 自分自身
- 部下や後輩
- 将来一緒に働くかもしれない人
- 考え方を共有したい相手

そういった **距離の近い誰か** に向けて、  
技術や判断の背景を含めて伝える場として使っています。

ZennやQiitaが「拡散」に強い場だとすると、  
このブログは「伝達」に向いた場、  
そんな位置づけです。

---

## おわりに

簡単だけど、地味に面倒で、  
これまで後回しにしていたことが、  
AIのおかげで一気に現実的になりました。

このブログは、AIコーディング時代の  
**Knowledge as Code の実践**として、  
そして、自分の知識や考えを伝えていくための場として、  
しばらく運用してみるつもりです。

同じように、  
アウトプットの置き場や、  
知識の伝え方に悩んでいる人にとって、  
ひとつの参考になれば嬉しいです。
